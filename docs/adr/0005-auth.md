# ADR-0005: Auth — JWT in httpOnly SameSite=Lax cookie, bcrypt cost 12, role-based guards

## Status

`Accepted`

## Date

2026-04-23

## Context

The app has exactly two principals: `CUSTOMER` and `ADMIN`. Customers own their own orders and can read/write them; admins manage the menu and order statuses. Sessions must survive SSR (server-side fetch needs the cookie), must not be readable by JavaScript (XSS containment), must flow through the Socket.IO handshake for the realtime gateway, and must play well with the customer-admin boundary in the same browser (same origin during demo).

Playbook reference: `docs/_playbook/05-auth-security.md` (decision tree: "same-origin SPA + SSR + realtime → JWT in httpOnly cookie > Authorization header").

## Options considered

### Option A — JWT (HS256) in httpOnly SameSite=Lax cookie, per-request validation

- **Pros:** Not reachable from XSS-injected JavaScript; sent automatically on SSR server-side fetch (with `withCredentials`); sent automatically on Socket.IO `io()` handshake (with `withCredentials: true`); stateless — no Redis or session store needed; SameSite=Lax blocks CSRF on idempotent cross-site requests, and CSRF double-submit token covers state-changing routes; bcrypt cost 12 gives ~200ms per login, acceptable friction.
- **Cons:** Can't revoke mid-lifetime (acceptable with short TTL — 1h access, no refresh for demo); HS256 (shared secret) means the secret must be present both in api and web-ssr if ssr needs to verify; we side-step that by never verifying on the ssr — the api is the only verifier, and the ssr forwards the cookie to the api on every data fetch.
- **Cost / effort:** moderate — requires cookie-parser, passport-jwt configured with a cookie extractor, a CSRF filter on mutating routes.
- **Reversibility:** high — swap cookie extractor for header extractor, or switch algorithm, without changing controllers.

### Option B — Session cookies backed by Redis (express-session or @fastify/session)

- **Pros:** Server-side revocation, "log out everywhere" is trivial, stateless clients.
- **Cons:** Adds Redis to the compose stack for a demo; session-store latency on every request; zero upside for a portfolio whose story is "modern stateless auth with SSR"; increases surface area without adding a checkbox to the README.
- **Cost / effort:** extra container, extra provider module, one more `.env` variable.
- **Reversibility:** medium — switching to JWT later means changing every request pipeline.

### Option C — OAuth / social login via NextAuth.js equivalent

- **Pros:** no password storage, recognizable UX.
- **Cons:** requires provider setup (Google OAuth consent screen etc.) that gates the demo behind external config; obscures the hand-rolled auth module that is itself a portfolio showpiece; "MFA" and "social login" are already explicitly out of scope in the PRD.
- **Cost / effort:** reducing my code, increasing vendor config cost.
- **Reversibility:** high — layer it on top of Option A later.

## Decision

JWT HS256 in httpOnly, SameSite=Lax, Path=/, Secure=false (dev) / Secure=true (prod) cookie, 1-hour TTL, no refresh token. Password hashing via bcrypt cost 12. RBAC via a custom `@Roles(...)` decorator + `RolesGuard` per controller method. CSRF on state-changing routes via the double-submit cookie pattern (client reads a non-httpOnly `csrf` cookie and mirrors it into the `x-csrf-token` header).

## Consequences

### Positive

- **SSR works:** Angular server-side fetches forward the `Cookie` header from the incoming request to the api; api decodes the JWT cookie; no special ssr-side auth logic.
- **Socket.IO works:** `io('/orders', { withCredentials: true })` sends the cookie on handshake; gateway's `handleConnection` reads `client.handshake.headers.cookie`, parses the JWT, attaches `user` to `client.data`; unauthenticated → `client.disconnect(true)`.
- **RBAC is declarative:** `@Roles('admin') @UseGuards(JwtAuthGuard, RolesGuard) deleteMeal(...)` — reads like the PRD describes it.
- **Passwords are salted + slow enough:** bcrypt cost 12 ≈ 200–300ms on the target dev machine; rate-limiting login is a future concern (out of MVP scope but called out in the threat model).

### Negative / trade-offs we accept

- Can't invalidate a leaked token before its 1-hour TTL. Acceptable for a demo. In production would add a `jti` blacklist in Redis.
- JWT shared secret (`JWT_SECRET`) must be present in `.env` — gitignored; `.env.example` documents the shape.
- SameSite=Lax lets top-level GET navigations carry the cookie, which is fine (GETs are read-only); CSRF only matters on POST/PUT/PATCH/DELETE and the double-submit token closes that.

### What this decision forces us to do

- [x] `apps/api/src/auth/` module with `auth.controller.ts` (register, login, me, logout), `auth.service.ts` (bcrypt compare, sign JWT, set cookie), `jwt.strategy.ts` (cookie extractor → `ExtractJwt.fromExtractors([(req) => req.cookies?.session])`), `roles.decorator.ts`, `roles.guard.ts`, DTOs (`RegisterDto`, `LoginDto`) with `class-validator`.
- [x] `main.ts`: `app.use(cookieParser())`, `app.useGlobalPipes(new ValidationPipe({ whitelist: true, transform: true }))`, CORS with `credentials: true, origin: ['http://localhost:4000', 'http://localhost:4200']`.
- [x] `AuthInterceptor` on the Angular side always sets `withCredentials: true`.
- [x] `CSRF` double-submit: any `POST/PUT/PATCH/DELETE` must include `x-csrf-token` header mirroring the `csrf` cookie (set at login); `CsrfGuard` on state-changing routes enforces.

## Revisit trigger

- Revisit if the app needs cross-origin clients (mobile, third-party) — switch to Authorization header with PKCE; if session revocation becomes a hard requirement; if compliance requires MFA (HIPAA, SOC2).
